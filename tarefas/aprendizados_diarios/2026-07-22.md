# Aprendizado Diário — 2026-07-22

## Missão Executada
**Correção P1 — Bloqueio MT5 e Startup do Operador LEON**

Classificação: OPERACIONAL LEON + INFRAESTRUTURA + MT5

## Decisões Tomadas
1. **rpyc 6.0.2 > rpyc 5.2.3**: Servidor wine MT5 roda rpyc 6.0.2. Cliente venv foi downgrade para 5.2.3 pelo mt5linux. Solução: reinstalar rpyc==6.0.2. mt5linux declara `rpyc==5.2.3` como dependência mas funciona com 6.0.2.
2. **systemd --user**: Usuário `leon` não tem sudo. Serviço instalado como `systemd-run --user --unit=leon-operator` (transient, não persistente após reboot). Alternativa: usar crontab @reboot ou solicitar configuração de sudo NOPASSWD.
3. **Autonomy scope**: Config.ini tinha `scope = execution`, código exige `scope = demo_execution` ou `learning_and_demo`. Corrigido em config.ini e `autonomy_state.json`.
4. **Telegram via .env**: Token e chat_id do Telegram não estão em config.ini (estão vazios). Estão em `/opt/leon/config/.env` com symlink de `/opt/leon/app/.env`.

## Erros Encontrados
1. **Conexão MT5 falha**: `ValueError: invalid message type: 18` — incompatibilidade rpyc 5.2.3 (cliente) vs 6.0.2 (servidor wine).
2. **Autonomy scope incorreto**: Config `scope = execution` bloqueia execução demo. Código espera `demo_execution` ou `learning_and_demo`.
3. **State file precedence**: `autonomy_state.json` sobrescreve config.ini para scope. Mesmo alterando config.ini, state file persistia valor antigo.
4. **sudo não disponível**: Usuário `leon` não está no sudoers. Serviços systemd não podem ser instalados como system service.

## Correções Aplicadas
| Arquivo | O quê |
|---------|-------|
| `config.ini` | `scope = execution` → `scope = demo_execution` |
| `data/autonomy_state.json` | `"scope": "execution"` → `"scope": "demo_execution"` |
| `/opt/leon/venv/` | rpyc reinstalado de 5.2.3 → 6.0.2 |
| `systemd --user` | Serviço leon-operator iniciado como transient unit |

## Padrões Identificados
- **rpyc version mismatch**: mt5linux (1.0.3) força rpyc==5.2.3, mas servidor wine MT5 usa rpyc 6.0.2. Sempre reinstalar rpyc==6.0.2 após instalar mt5linux.
- **Autonomy scope**: sempre verificar que `scope` no state file corresponde ao config.ini — state file tem precedência.
- **Sem sudo**: usuário `leon` não tem sudo. Usar `systemd-run --user` para serviços transient, ou crontab para persistência.

## Recomendações para Outros Agentes
1. **Após instalar mt5linux**: executar `pip install 'rpyc>=6.0,<7.0'` para restaurar compatibilidade.
2. **Ao alterar config.ini**: verificar se módulo lê direto do config.ini ou de state files cached.
3. **Para persistência do operador após reboot**: criar entrada crontab com `@reboot systemd-run --user --unit=leon-operator /opt/leon/venv/bin/python /opt/leon/app/src/leon_operator.py`.

## Segunda Missão — 2026-07-22

**Implementação de Correções ABC no Operacional LEON**

Classificação: OPERACIONAL LEON + ARQUITETURA

### Decisões Tomadas
1. **Detecção ABC aditiva**: Implementada como novo detector separado (`detect_abc_correction`) que é chamado dentro de `analyze_elliott_context()` apenas quando todas as detecções de impulso/Fibonacci falham. O catch-all `"CORRECAO"` é preservado como fallback.
2. **4 pivots (w0, a, b, c)**: A detecção correta de ABC exige conhecer o início da onda A (w0), não apenas os 3 pivots A-B-C. A função `_abc_quad` extrai 4 pivots alternados.
3. **Campos InterestZone**: Os campos `scenario_present`, `possible_wave_c`, `correction_possible`, `correction_detected`, `correction_subtype` agora são populados em todos os caminhos do `analyze_elliott_context()`, não apenas no ABC.

### Erros Encontrados e Corrigidos
1. **`_abc_pivot_trio` com 3 preços insuficiente**: A versão inicial usava 3 preços (a, b, c) mas não conseguia calcular corretamente o comprimento da onda A (precisava do preço anterior a "a"). Corrigido para 4 preços (w0, a, b, c).
2. **start_offset=-6 muito restritivo**: A busca por padrão ABC a partir de offset fixo impedia detecção em listas pequenas de pivots. Substituído por busca reversa completa em toda a lista.

### Padrões Identificados
- Detecção ABC deve ser SEMPRE aditiva ao pipeline existente (impulso → Fibonacci → onda 3 → ABC → catch-all)
- ABC requer 4 pontos de preço para classificação correta de subtipo
- `RANGE` é um subtipo ABC válido (correção curta), mas com confiança baixa

### Arquivos Modificados
| Arquivo | Mudança |
|---------|---------|
| `src/institutional_analysis_engine.py` | +203 linhas: `_abc_quad()`, `_abc_subtype()`, `detect_abc_correction()`, integração em `analyze_elliott_context()` + campos InterestZone |
| `src/teste_institutional_analysis.py` | +251 linhas: 20 novos testes (AbcQuadTests, AbcSubtypeTests, DetectAbcCorrectionTests, AnalyzeElliottContextAbcTests) |

### Recomendações
1. Ao implementar detecção de padrões de preço, sempre pensar em quantos pontos de dados são necessários para calcular as propriedades do padrão (ex: ABC precisa de 4 pontos, não 3).
2. A ordem de precedência no `analyze_elliott_context()` é crítica: Fibonacci > Impulso completo > Onda 3 parcial > ABC estruturado > catch-all genérico.

## Terceira Missão — 2026-07-22 (Multigrau)

**Expansão da detecção ABC para M15 (micro-correções)**

Classificação: ARQUITETURA + OPERACIONAL LEON

### Decisões Tomadas
1. **`analyze_elliott_context()` é timeframe-agnostic**: A função aceita qualquer array de candles. Basta chamá-la com `candles_m15` para obter detecção ABC no M15.
2. **Mantido separado do H1**: O label M15 (`elliott_m15`) não substitui o label H1 (`elliott`). Ambos coexistem. O H1 continua sendo o principal para gates de entrada.
3. **Direção do M15 herdada do H1**: `tendencia_m15 = elliott_context.get("direction") or tendencia_elliott` — a micro-análise respeita o contexto do timeframe superior.
4. **Exibição dedicada**: Console mostra seção "ELLIOTT M15 (MICRO ESTRUTURA)" separada da seção H1.
5. **Sem novos gates**: M15 ABC é informativo, não cria gates de entrada. Todo o fluxo de decisão (`setup_onda_confirmado`, `smc_guard`, `top_down_confirmado`) permanece inalterado.

### Padrões Identificados
- `analyze_elliott_context()` pode ser chamado com qualquer timeframe — não tem hardcoding de resolução.
- A detecção ABC funciona em M15 com 220 candles, gerando pivots mais granulares que no H1.
- Análise multigrau não requer novas engines — apenas reuso da mesma função com dados diferentes.

### Arquivos Modificados
| Arquivo | Mudança |
|---------|---------|
| `src/leon.py` | +30 linhas: chamada `analyze_elliott_context(candles_m15, ...)`, variável `elliott_m15`, seção de display M15 |

### Recomendações
1. Não confundir `elliott_m15` com o label principal — cada um opera em seu timeframe.
2. M15 ABC pode gerar mais sinais que H1 (mais pivots). Isso é esperado e desejável — o olho no micro ajuda a enxergar entradas dentro das pernas maiores.

## Quarta Missão — 2026-07-22 (Backtest + Ajustes)

**Backtest de resultados e ajustes no operador baseado em evidência**

Classificação: OPERACIONAL LEON + ARQUITETURA + DESEMPENHO

### Diagnóstico (Dados de 2026-07-22)
- **75 registros** no CSV (73 fechados)
- **Winrate 5.5%** (4W / 69L)
- **100% COMPRA** num mercado que caiu 28 pontos
- **SMC sempre NEUTRO**, **todos SETUP FRACO**
- **Nunca detectou VENDA** — só operou compra em tendência de queda

### Causas Raiz Identificadas
1. **Top-down `_bias()` primitivo**: Comparava apenas primeiro vs último close. Perdia toda estrutura intermediária. O fallback mult-TF usava o mesmo CSV para todos os timeframes.
2. **`_displacement()` threshold 1.35x excessivo**: Exigia corpo 35% acima da mediana, poucos candles qualificavam. Causa direta do SMC sempre NEUTRO.
3. **POI mínimo 70 muito alto**: POI score começa em 25, precisava de 3-4 condições simultâneas para atingir 70. Mesmo com FVG válido, SMC ficava NEUTRO.
4. **Confiança sem feedback**: Thresholds fixos (90/70/50). Batch review gerava recomendações mas `applied_automatically = False` sempre. Nenhum aprendizado a partir dos resultados.

### Ajustes Implementados (autorizados pelo usuário)
| Ajuste | O quê | Arquivo | Risco |
|--------|-------|---------|-------|
| 1 | `_bias()` reescrito com SMA5/SMA20 + 0.2% threshold + HH/HL fallback | `top_down_agent.py:70-98` | Baixo |
| 2 | Fallback M15: 4→8 candles, H1: 16→24, H4: 48→48, macro: 120→120 (janelas ajustadas para o novo `_bias`) | `top_down_agent.py:96-99` | Baixo |
| 3 | `_displacement` threshold: 1.35x → 1.15x | `institutional_analysis_engine.py:241` | Baixo |

| 6 | `calcular_confianca_ajustada()` + wire em `leon.py` | `confidence_engine.py:41-65`, `leon.py:479-486` | Baixo |

### Resultados dos Testes

### Segurança
- Nenhum guard removido
- SMC entry guard permanece intacto (ainda exige BULLISH/BEARISH)
- Conta real permanece bloqueada
- Nenhuma alteração em MT5, risco, TP/SL, ou execução

### Padrões Identificados
- `_bias()` baseado em médias curta/longa é mais robusto que comparação first/last
- Thresholds de confirmação SMC precisam ser testados com dados reais para calibrar
- Feedback loop de confiança deve ser batch-agnostic (acumula ajustes ao longo do tempo)
- Ajustes de confiança devem respeitar limites [0, 100] para não inverter o score
